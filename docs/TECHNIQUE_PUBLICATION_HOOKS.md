# Technique Publication Hooks

This note defines how technique-sensitive hooks should land without making techniques behave like auto-activated skills.

## Core rule

Techniques should not grow through runtime auto-activation hooks.

A technique is public practice meaning, not a hidden execution plugin.

Technique hooks should therefore live at publication time, evidence time, and review time.

## Good hook moments

These moments are narrow and reviewable:

- a closeout packet survives into a reusable donor note
- a live receipt is published
- a cross-layer technique candidate intake surface changes
- a promotion-readiness surface changes
- a canonical review note changes
- an external evidence ledger adds one more bounded donor signal
- a neighboring layer proves the same pattern in another stable context

These are honest moments to emit observations.

## Avoid this anti-pattern

Do not create a runtime hook that says:

- "the model used a pattern once, therefore a technique now exists"
- "this prompt smells like a technique, auto-canonicalize it"
- "technique metadata says high maturity, therefore promote now"

That path turns advisory metadata into hidden judgment.

## Recommended technique signals

Examples of honest observation signals:

- `cross_layer_candidate_seen`
- `distillation_pressure_observed`
- `second_consumer_confirmed`
- `cross_context_evidence_present`
- `promotion_readiness_changed`
- `overlap_hold_open`
- `canonical_review_blocked`

These signals still need beacon rules and review surfaces. They are not self-authorizing.

## Why this matters

Techniques need stable extraction pressure, not accidental activation pressure.

The right arc is:

`live use -> bounded receipt or harvest -> observation -> beacon -> owner review surface -> reviewed distillation or promotion`
